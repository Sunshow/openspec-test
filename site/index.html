What is OpenSpec?

一种让规范成为开发核心驱动力的方法论

OpenSpec 提出了 SDD（Spec-Driven Development，规范驱动开发） 方法论。它的核心理念是：以结构化的规范（Spec）作为需求、设计和实现之间的唯一真实来源（Single Source of Truth），让团队在同一套语言下协作。

不同于传统的文档驱动或口头沟通，OpenSpec 将每一个行为期望都定义为可追踪、可验证的 Spec 条目，确保从需求提出到代码交付的全链路可追溯。

核心原则：规范先行，代码跟随。每一行代码都有对应的 Spec 定义，每一个 Spec 都可以被验证和追踪。

核心概念

OpenSpec 围绕五个关键概念构建完整的规范驱动开发体系

Spec（规范）

以 Given/When/Then 格式定义的行为规范，是需求的最小可验证单元，也是整个体系的基石。

Change（变更）

一次有意义的功能变更单元，包含提案、规范增量、设计方案和任务清单的完整集合。

Proposal（提案）

变更的起点，说明为什么要做、做什么、影响范围，为后续的规范和设计提供上下文。

Design（设计）

技术方案文档，描述如何实现 Spec 中定义的行为，包含架构决策、数据流和文件变更清单。

Tasks（任务）

将设计方案拆解为可执行的实施任务清单，每个任务对应具体的代码变更，可逐项跟踪完成状态。

工作流程

从提案到归档，六个步骤完成一次规范驱动的开发迭代

1

Proposal

提出变更提案，明确目标与范围

2

Delta Specs

定义新增或变更的行为规范

3

Design

编写技术方案与架构设计

4

Tasks

拆解为可执行的任务清单

5

Implementation

逐项实施任务，编写代码

6

Archive

归档变更，合并到基线规范

快速上手

只需几步，即可在项目中引入 OpenSpec 规范驱动开发流程

1 项目目录结构

# 在项目根目录创建 openspec 结构 openspec/ ├── specs/ │ └── baseline.spec.md # 基线规范 └── changes/ └── my-feature/ # 一次变更 ├── proposal.md # 提案 ├── specs/ │ └── delta.spec.md # 增量规范 ├── design.md # 技术设计 └── tasks.md # 任务清单

2 编写你的第一个 Spec

# delta.spec.md ## ADDED: SPEC-001 — 用户登录 > Given 用户在登录页面 > When 输入正确的用户名和密码 > Then 成功跳转到首页 > And 显示用户头像和昵称
